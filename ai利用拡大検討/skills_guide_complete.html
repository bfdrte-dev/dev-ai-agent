社内ドキュメント｜Claude Code 活用

Claude Code Skills 入門ガイド AIに「自分たちの仕事のやり方」を覚えさせる仕組み

なぜ毎回お膳立てが必要なのか

Claude Code は非常に賢いツールです。ただ、「私たちのチームの仕事のやり方」は知りません。

これは性能の問題ではなく、仕組みの話です。Claude は新しいセッションを開くたびに、まっさらな状態からスタートします。

よくある状況原因
毎回フォーマットを手打ちで指定しているClaude は前回の指示を覚えていない
「違う、こっちの書き方で」と何度も修正しているチーム固有のルールを知らない
AI を使っているのに、準備で疲れてしまう毎回ゼロからセットアップしているから
図 — Skills なし：毎回ゼロからのセットアップ

この仕組みが引き起こす問題は3つあります。

問題1：一貫性がない

同じ「週次レポートを作って」という依頼でも、今日は詳細版、明日は箇条書き3行。フォーマットがセットされていないので、毎回 Claude が独自に判断するしかありません。

問題2：チーム固有の判断基準が入っていない

Claude は汎用的には優秀です。でも「うちの会社では結論から書くのがルール」「上司は3ページ以上の資料を読まない」といった、チーム環境でしか通用しないルールは知りません。

問題3：セットアップの手間が積み重なる

書式の指定を手打ちして、判断基準を説明して、前提情報を共有して——この「お膳立て」に、1日でどれだけの時間を使っているでしょうか。

図 — 3つの問題を Skills が解決する

Skills とは何か

一言で言うと：「AIに仕事のやり方をセットするファイル」です。

ポケモンの「わざマシン」をイメージしてください。ディスクを1枚読み込ませるだけで、そのポケモンは新しい技を覚えます。覚えた技は、毎バトル何度でも使えます。忘れさせない限り、ずっとです。

図 — Skills は Claude の「わざマシン」

SKILL.md というファイルを1つ作って Claude にセットします。すると Claude は、セッションを開くたびにその指示に従って動きます。毎回お膳立てする必要がなくなります。

図 — Skills なしとありの違い
Skills の本質は「作業の自動化」ではありません。
「チームの判断基準を言語化して、AI にセットする行為」です。「結論から書く」「データは3点以内にする」——これは作業手順ではなく、チームの判断基準です。判断の資産化。Skills を一言で表すなら、これに尽きます。

必要なのは SKILL.md というファイル1つ

Skills の仕組みは、思っているよりずっと簡単です。必要なのは SKILL.md というファイル1つだけ。中身は2つのパートに分かれています。

図 — SKILL.md の構造

パート1：「表紙」（name と description）

ファイルの先頭に、Skills の名前と「いつ使うべきか」の説明を書きます。Claude はこの表紙を見て、今のタスクにこの Skills を使うべきかどうかを判断します。ここが曖昧だと、せっかく作った Skills が起動しません。

パート2：「本文」（具体的な指示）

チームの仕事のやり方そのものです。「こういう順番で書いて」「このフォーマットに従って」「この点だけは必ず確認して」といった具体的な手順やルールを書きます。

実際のファイル例を見てみましょう。

---
name: 週次営業レポート
description: 毎週月曜に営業チームの週次レポートを作成するときに使う。上司の田中部長向け。
---

# 週次営業レポート

## フォーマット
- 1ページ目に結論（今週の着地見込み）を書く
- 数字は必ず前週比で記載する（例：先週比+12%）
- 案件はA/B/Cの3段階で分類する
- 課題は1つに絞り、来週の具体的なアクションを1行で添える

## 田中部長ルール
- 3ページ以上は読まない。必ず2ページ以内
- グラフより表を好む
- 「検討中」は書かない。必ず「いつまでに何をする」に変換する

「田中部長ルール」のような、職場でしか通用しないルールこそが Skills の価値です。Claude がどれだけ賢くても、田中部長の好みは教えないと分かりません。

起動方法は2通り

図 — Skills の起動方法

Skills を50個入れても Claude は遅くならない

「Skills をたくさん入れたら、Claude の動きが遅くなるんじゃないの？」——答えは、遅くなりません。Claude は全ての Skills を一度に読み込むのではなく、必要なものだけを段階的に読み込む仕組みになっています。

図 — 段階的な読み込みの仕組み
表紙の「説明」が曖昧だと、レベル1の段階で Claude は「今回は関係なさそう」と誤判断してスルーします。「いつ、どんなタスクで使うべきか」を明確に書くのが最重要です。

初めての Skills は6ステップで完成する

まず Skills を作るべきタイミングを見極めてください。以下の3つのうち1つでも当てはまれば Skills の出番です。

  • 手打ちしている前提情報がある（書式、ルール、トーンなど）
  • 毎回同じクオリティで出してほしいタスクがある
  • いつも忘れるステップがある
図 — 6ステップの全体像

ステップ1：繰り返しの中に Skills の種がある

図 — 3回繰り返したら Skills の候補

今日から1週間、Claude に手打ちしている内容をメモしてみてください。3回以上繰り返しているものが見つかるはずです。

ステップ2：フォルダ1つとファイル1つで準備完了

Claude Code のプロジェクト内に以下の構成でファイルを作ります。

.claude/skills/
  └── weekly-report/
        └── SKILL.md

ステップ3：最初は5行で十分

図 — 完璧なSkillsより動くSkillsが価値ある

最初はこれだけで十分です。

---
name: 週次営業レポート
description: 毎週月曜に営業チームの週次レポートを作成するときに使う
---

結論を最初に書く。数字は前週比。課題は1つに絞る。2ページ以内。

本文はたった1行です。使っていくうちに「田中部長はグラフより表が好きだな」と気づいたら、その都度追記していけばいいんです。

ステップ4：比較テストで効果を確認

同じ依頼を、Skills をオンにした状態とオフにした状態の両方で試してください。差がなければ「Claude が元々知っていること」しか書いていない可能性があります。チームにしか通用しない判断基準を追加しましょう。

ステップ5：失敗するたびに Skills は育つ

「ここがまだ違うな」と感じた部分を、その都度 SKILL.md に追記します。

図 — Skillsは使いながら育てるもの
同じ修正を3回したなら、それを SKILL.md に書き足しましょう。

ステップ6：Claude 自身が改善案を出してくれる

Anthropic が公式で提供している skill-creator という Skills を使うと、Claude 自身が「この Skills をもっと良くするにはこうしたらどうですか」と提案してくれます。AI に仕事のやり方をセットしたら、今度は AI が「こうしたほうがもっと効率いいですよ」と返してくる——この循環が Skills の面白さです。

使える Skills に育てるための3つの危険信号

同じタスクを3回実行して、毎回違う結果が出るなら、その Skills は改善が必要です。

図 — 曖昧より具体、一般より固有
危険信号1：指示が無視される
「分かりやすく書いて」は曖昧です。「結論を最初に書く。1文は40文字以内。専門用語は使わない。」——数字や具体的なルールで書くのがコツです。
危険信号2：Skills が起動しない
表紙の description に問題があるケースです。「いつ、どんなタスクで使うべきか」が一読で分かる説明になっているか確認してください。
危険信号3：Skills ありとなしで差がない
Skills に「Claude が元々知っていること」しか書いていない状態です。チームしか知らない判断基準を書くのが正解です。

ベストプラクティス 4 原則

図 — Skills 運用の4原則（下が基礎）

① Claude は賢い。知っていることを教える必要はない。
「JSON 形式で出力して」のような一般的な指示は不要です。あなたの仕事の「なぜそうするか」「どの順番でやるか」という判断基準を教えてください。

② SKILL.md は100行前後に保つ。
大きくなりすぎると Claude の判断の質が下がります。Skills が大きくなりすぎたら、参考資料は別のフォルダに分けて格納しましょう。SKILL.md 本体は指示の核だけに集中させるのが長く使うコツです。

③ 理想ではなく、実際のケースでテストする。
架空の依頼で試しても意味がありません。月曜日に実際にやる仕事で Skills を使って試してみてください。リアルな仕事でこそ、不足している指示が見えてきます。

④ 完璧を目指すより、使いながら育てる。
3回同じ修正をしたら SKILL.md に追記する。この繰り返しが、チーム専用の Skills に育てていく最短ルートです。

もっと深く使いたい人へ（応用編）

すぐに使える公式 Skills がある

図 — Anthropic 公式提供の Skills（GitHub からインストール可）

2種類の置き場所がある

場所用途
ホームフォルダの ~/.claude/skills/自分だけが使う個人用 Skills。どのプロジェクトでも使える
プロジェクト内の .claude/skills/バージョン管理に含まれる。チーム全員が同じ Skills を使える
チーム共有こそが Skills の隠れた本質です。
あなたの判断基準がファイルになるということは、チームのメンバーが入れ替わっても、仕事の品質が維持されるということです。判断基準が人に依存しなくなる——組織にとってかなり大きな変化です。

まとめ：6つのポイント

  • Skills は「AIに仕事のやり方をセットするファイル」。ポケモンのわざマシンのように、一度セットすれば毎回発動します
  • Claude は賢いですが「チームの仕事のやり方」は知りません。Skills がなければ、毎回ゼロからお膳立てすることになります
  • 必要なのは SKILL.md というファイル1つ。中身は「名前＋説明＋指示」だけです
  • 50個入れても大丈夫です。Claude は必要な Skills だけを選んで読み込みます
  • 小さく始めて育てるのが正解です。5行の動く Skills は80ルールの完璧な Skills に勝ります
  • Skills の本質は作業の自動化ではなく「判断の資産化」。チームの判断基準を言語化して AI にセットする行為です
